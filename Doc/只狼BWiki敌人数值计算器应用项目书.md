# 只狼 BWiki 敌人数值计算器应用项目书

## 1. 项目背景与建设目标

### 1.1 项目背景

根据 [`Doc/项目需求.md`](Doc/项目需求.md) 的定义，本项目面向 BWiki 托管环境，目标是在基于 [`MediaWiki`](Doc/项目需求.md:9) 的页面中制作可嵌入的数据应用。现阶段三个方向中，优先级最高且本期正式实施的是“数据计算器”应用，即根据敌人的基础属性与多类加成规则，动态计算并展示敌人的血量、躯干、躯干恢复等关键数值。

根据 [`Doc/敌人数据结构和计算逻辑.md`](Doc/敌人数据结构和计算逻辑.md)，该应用并非普通静态表格，而是一个带有明确业务规则、复杂 buff 组合逻辑、条件切换、表格裁剪与明细追踪能力的数据型工具。它的使用场景不是独立官网或专题页，而是 Wiki 条目中的工具附表，因此必须兼顾以下要求：

1. 与 Wiki 正文风格兼容。
2. 计算逻辑准确且可解释。
3. 数据来源便于持续更新。
4. 托管方案稳定、免费、易维护。

### 1.2 建设目标

本项目的建设目标如下：

1. 制作一个可嵌入 BWiki / [`MediaWiki`](Doc/项目需求.md:9) 页面中的敌人数值计算器。
2. 基于敌人基础数据、天色、周目、难度、阶段和特殊 buff 规则，自动计算目标数值。
3. 以表格形式展示计算结果，并根据敌人类型与规则动态调整列与行的显示结构。
4. 支持点击单元格后展示详细 tip，说明当前数据使用了哪些基础值、buff、ID 与倍率。
5. 将原始 Excel 数据整理为前端可直接消费的数据文件，并托管到外部稳定地址。
6. 建立一套后续可持续扩展到其他应用的基础数据与前端架构。

### 1.3 项目范围

本期项目范围聚焦于 [`Doc/项目需求.md`](Doc/项目需求.md:37) 中定义的“应用二：数据计算器”，具体包括：

- 敌人 HP 上限计算
- 敌人躯干上限计算
- 敌人躯干恢复计算
- 多条件切换下的动态表格展示
- 单元格 tip 明细展示
- 数据整理、外部托管与 Wiki 页面嵌入方案

### 1.4 本期与后续期边界

本期**不纳入**以下内容：

1. [`Doc/项目需求.md`](Doc/项目需求.md:16) 中提到的“基础数据展板”。
2. [`Doc/项目需求.md`](Doc/项目需求.md:48) 中提到的“伤害计算器”。
3. 与本期核心需求无直接关系的其他历史文件、临时文件或扩展表。

## 2. 使用场景与目标用户

### 2.1 Wiki 读者查询场景

普通读者希望在敌人条目页或相关附表页中，快速查询某个敌人在指定周目、天色、难度、阶段下的生命值、躯干值与躯干恢复值。该场景要求：

- 进入页面即可使用，无需额外安装环境。
- 展示结构直观，符合 Wiki 阅读习惯。
- 不同条件切换后的结果能即时刷新。

### 2.2 攻略编写与校对场景

攻略作者或数据整理者需要核对多种条件下的数值，并比对不同难度、天色、再战、死斗模式下的差异。该场景要求：

- 表格展示完整、规整。
- 规则分支处理清晰，避免误读。
- 能够快速定位某个结果的来源。

### 2.3 数据维护与规则核对场景

维护者需要确认某个敌人条目数据是否正确，尤其是在涉及特殊 buff、阶段切换 buff、再战/死斗/连战等场景时，需要追踪具体乘算链。该场景要求：

- tip 明细足够详细。
- 每个结果应可反查基础值、buff 名称、buff ID 与倍率。
- 数据与规则结构便于后续维护和修正。

## 3. 需求分析

### 3.1 输入条件

根据 [`Doc/敌人数据结构和计算逻辑.md`](Doc/敌人数据结构和计算逻辑.md) 的定义，前端至少需要支持以下输入条件：

1. 敌人选择
2. 周目选择
3. 天色选择
4. 阶段选择
5. 难度维度展示
6. 不同模式维度切换，包括主线、再战、死斗，以及怨鬼特有修罗连战

其中：

- 天色序列由 [`time_seq`](Doc/敌人数据结构和计算逻辑.md:48) 控制。
- 阶段下拉项数量由 [`phase_num`](Doc/敌人数据结构和计算逻辑.md:57) 决定。
- 难度维度需覆盖普通、钟鬼、交符、双难，且“敲钟”和“钟鬼”为同义词，见 [`Doc/敌人数据结构和计算逻辑.md`](Doc/敌人数据结构和计算逻辑.md:104)。

### 3.2 输出结果

本项目最终输出的核心数据为：

1. HP 上限
2. 躯干上限
3. 躯干恢复

所有计算结果统一执行向下取整，要求与 [`Doc/敌人数据结构和计算逻辑.md`](Doc/敌人数据结构和计算逻辑.md:217) 一致。

### 3.3 表格展示要求

根据 [`Doc/敌人数据结构和计算逻辑.md`](Doc/敌人数据结构和计算逻辑.md:62) 与 [`DB/表格布局示例.xlsx`](DB/表格布局示例.xlsx)，结果展示并非固定表结构，而是要根据敌人和场景动态裁剪。

主要规则包括：

1. 当 [`base_bonus_buff`](Doc/敌人数据结构和计算逻辑.md:20) 长度为 1，或敌人为“赤鬼（虎口阶梯）”时，不显示钟鬼与双难。
2. 当敌人类型不是 boss 时，不显示再战、连战和死斗。
3. 当 [`base_bonus_buff`](Doc/敌人数据结构和计算逻辑.md:20) 长度为 2 且满足指定天色条件时，普通与钟鬼合并显示，交符与双难合并显示。
4. 当敌人为“怨恨之鬼”时，显示连战数据。
5. 当敌人为 boss 但不是怨恨之鬼时，不显示连战列。
6. 当敌人名称中包含“心中”时，只显示再战和死斗两列数据。

### 3.4 tip 栏说明要求

tip 栏属于本项目的关键功能，而不是可选附属项。根据 [`Doc/敌人数据结构和计算逻辑.md`](Doc/敌人数据结构和计算逻辑.md:187)，当用户点击某个结果单元格时，需展示：

- 基础值名称与数值
- 参与计算的 buff 名称
- 参与计算的 buff ID
- 对应倍率值
- 最终完整乘算链

展示格式应尽量靠近 [`Doc/敌人数据结构和计算逻辑.md`](Doc/敌人数据结构和计算逻辑.md:197) 所给示例。

### 3.5 样式与宿主兼容要求

根据 [`sumary/frontdemo_design_summary.md`](sumary/frontdemo_design_summary.md)，应用必须满足以下样式要求：

1. 黑白灰体系为主。
2. 弱装饰、弱色彩，不抢正文层级。
3. 主体为规整表格，信息密度高且可读。
4. 控件更接近表单而不是“产品按钮”。
5. 桌面端优先完整显示，小屏端允许横向滚动降级。

## 4. 业务规则解读

### 4.1 敌人数据结构

根据 [`Doc/敌人数据结构和计算逻辑.md`](Doc/敌人数据结构和计算逻辑.md:1)，每个敌人至少包含以下业务字段：

- `name`
- `select_menu` 和 `npc_param` 的 id
- `base_hp`
- `base_stamina`
- `base_stamina_recover`
- `base_bonus_buff`
- `base_bonus_buff_NG1_no_charm`
- `multi_NG_balance_buff`
- `phase_change_buff`
- `special_buff`
- `mind_battlefield_buff`
- `re_battle_buff`
- `death_battle_buff`
- `enemy_type_buff`
- `time_seq`
- `NG_buff`
- `phase_num`

其中，敌人数据既承担“基础数值载体”的作用，也承担“后续 buff 链接入口”的作用。

### 4.2 buff 类型与作用范围

项目中涉及的 buff 可分为以下几类：

1. 区域天色加成类
   - [`base_bonus_buff`](Doc/敌人数据结构和计算逻辑.md:20)
   - [`base_bonus_buff_NG1_no_charm`](Doc/敌人数据结构和计算逻辑.md:23)
   - [`multi_NG_balance_buff`](Doc/敌人数据结构和计算逻辑.md:26)

2. 阶段与特殊类
   - [`phase_change_buff`](Doc/敌人数据结构和计算逻辑.md:29)
   - [`special_buff`](Doc/敌人数据结构和计算逻辑.md:34)

3. 模式类
   - [`mind_battlefield_buff`](Doc/敌人数据结构和计算逻辑.md:38)
   - [`re_battle_buff`](Doc/敌人数据结构和计算逻辑.md:39)
   - [`death_battle_buff`](Doc/敌人数据结构和计算逻辑.md:40)
   - 怨鬼修罗连战专用 buff

4. 兵种类
   - [`enemy_type_buff`](Doc/敌人数据结构和计算逻辑.md:44)

5. 周目成长类
   - [`NG_buff`](Doc/敌人数据结构和计算逻辑.md:52)

### 4.3 buff 列表推导规则

#### 4.3.1 区域天色加成类列表推导

根据 [`Doc/敌人数据结构和计算逻辑.md`](Doc/敌人数据结构和计算逻辑.md:84)，[`base_bonus_buff`](Doc/敌人数据结构和计算逻辑.md:20) 只填写序列起始 ID，完整 list 需要根据“下一行 ID 的尾号”推导，规则如下：

1. 下一行 ID 结尾既不是 1 也不是 4，则序列长度为 1。
2. 下一行 ID 结尾为 4，则序列长度为 2。
3. 下一行 ID 结尾为 1，则序列长度为 5，形式如 `7010, 7011, 7012, 7013, 7014`。

另外两个列表以 [`base_bonus_buff`](Doc/敌人数据结构和计算逻辑.md:20) 为锚定：

- [`base_bonus_buff_NG1_no_charm`](Doc/敌人数据结构和计算逻辑.md:23) = `base_bonus_buff + 700`
- [`multi_NG_balance_buff`](Doc/敌人数据结构和计算逻辑.md:26) = `base_bonus_buff + 500`

#### 4.3.2 再战、连战、死斗类列表推导

根据 [`Doc/敌人数据结构和计算逻辑.md`](Doc/敌人数据结构和计算逻辑.md:93)，下列 buff 统一长度为 4：

- [`mind_battlefield_buff`](Doc/敌人数据结构和计算逻辑.md:38)
- [`re_battle_buff`](Doc/敌人数据结构和计算逻辑.md:39)
- [`death_battle_buff`](Doc/敌人数据结构和计算逻辑.md:40)
- `shura_battlefield_buff`
- `shura_battle_buff`

即填写的起始 ID 加上其后的连续三个 ID，例如 `7950, 7951, 7952, 7953`。

### 4.4 一周目计算规则

根据 [`Doc/敌人数据结构和计算逻辑.md`](Doc/敌人数据结构和计算逻辑.md:114)，一周目计算规则如下：

1. 普通：基础值 × [`base_bonus_buff`](Doc/敌人数据结构和计算逻辑.md:20)
2. 钟鬼：基础值 × [`base_bonus_buff`](Doc/敌人数据结构和计算逻辑.md:20) 的钟鬼对应值
3. 交符：基础值 × [`base_bonus_buff_NG1_no_charm`](Doc/敌人数据结构和计算逻辑.md:23) × [`enemy_type_buff`](Doc/敌人数据结构和计算逻辑.md:44)
4. 双难：基础值 × [`base_bonus_buff_NG1_no_charm`](Doc/敌人数据结构和计算逻辑.md:23) 的钟鬼对应值 × [`enemy_type_buff`](Doc/敌人数据结构和计算逻辑.md:44)

### 4.5 多周目计算规则

根据 [`Doc/敌人数据结构和计算逻辑.md`](Doc/敌人数据结构和计算逻辑.md:123)，二至八周目需要在一周目基础上引入：

- [`multi_NG_balance_buff`](Doc/敌人数据结构和计算逻辑.md:26)
- [`NG_buff`](Doc/敌人数据结构和计算逻辑.md:52)

同时，根据 normal / charm 难度不同，周目加成索引分普通序列与交符序列两段。

### 4.6 再战与死斗计算规则

根据 [`Doc/敌人数据结构和计算逻辑.md`](Doc/敌人数据结构和计算逻辑.md:134) 与 [`Doc/敌人数据结构和计算逻辑.md`](Doc/敌人数据结构和计算逻辑.md:143)，再战与死斗都采用两层 buff 乘算：

- 场地 buff
- 对应模式 buff

注意其实际 buff 排列顺序是“普通、双难、交符、钟鬼”，因此前端调用顺序必须重排后再映射到表格展示顺序，见 [`Doc/敌人数据结构和计算逻辑.md`](Doc/敌人数据结构和计算逻辑.md:152)。

### 4.7 怨鬼修罗连战规则

根据 [`Doc/敌人数据结构和计算逻辑.md`](Doc/敌人数据结构和计算逻辑.md:156)，怨鬼具有独立的修罗连战计算逻辑，应使用：

- `shura_battlefield_buff`
- `shura_battle_buff`

且调用顺序与再战/死斗一样，需要适配表格中的难度顺序。

### 4.8 特殊加成与阶段切换

#### 4.8.1 阶段切换加成

[`phase_change_buff`](Doc/敌人数据结构和计算逻辑.md:29) 的特点是：

- key 表示生效阶段
- value 表示加成 ID
- 仅在对应阶段生效
- 不会保留到其他阶段

#### 4.8.2 特殊加成

[`special_buff`](Doc/敌人数据结构和计算逻辑.md:34) 对所有难度生效，但 `stamina_bonus` 可能与 `hp_bonus` 不同，因此计算 HP、躯干、躯干恢复时要允许采用不同倍率分支，见 [`Doc/敌人数据结构和计算逻辑.md`](Doc/敌人数据结构和计算逻辑.md:103)。

### 4.9 特殊显示规则

根据 [`Doc/敌人数据结构和计算逻辑.md`](Doc/敌人数据结构和计算逻辑.md:165)，当 [`base_bonus_buff`](Doc/敌人数据结构和计算逻辑.md:20) 的 list 长度为 2 且满足对应天色条件时，普通数据等于钟鬼数据，交符数据等于双难数据，因此表格层面应进行合并显示，而不是重复绘制四组完全独立的结果。

### 4.10 统一取整与输出规范

所有结果统一执行向下取整，且应当在：

1. 所有 buff 乘算完成后进行。
2. 表格展示与 tip 展示中保持一致。
3. 不因不同列或不同模式而改变取整规则。

## 5. 数据源分析与数据整理方案

### 5.1 原始数据来源说明

本项目的数据来源包括：

- [`DB/整合表.xlsx`](DB/整合表.xlsx)：核心业务数据总表。
- [`DB/表格布局示例.xlsx`](DB/表格布局示例.xlsx)：前端表格布局与控件位置参考。
- [`Doc/敌人数据结构和计算逻辑.md`](Doc/敌人数据结构和计算逻辑.md)：业务规则说明。
- [`sumary/frontdemo_design_summary.md`](sumary/frontdemo_design_summary.md)：视觉风格总结。

### 5.2 [`DB/整合表.xlsx`](DB/整合表.xlsx) 的角色定义

[`DB/整合表.xlsx`](DB/整合表.xlsx) 在整个项目中应被定义为“前端运行数据的母表”，其角色不是直接上线使用，而是作为整理与导出的上游源数据。

它承担以下职责：

1. 提供敌人的基础值。
2. 提供敌人与各种 buff ID 的对应关系。
3. 提供生成显示规则所需的敌人属性。
4. 提供 tip 中所需的名称、ID、加成值来源。

### 5.3 [`DB/表格布局示例.xlsx`](DB/表格布局示例.xlsx) 的角色定义

[`DB/表格布局示例.xlsx`](DB/表格布局示例.xlsx) 的角色是“界面结构蓝本”，重点不在数据，而在以下信息：

- 控件摆放位置
- 表头结构
- 表格层级
- 合并单元格风格
- 不同模式列的组织方式

它应作为前端布局设计依据，而不是作为前端数据源。

### 5.4 前端运行数据模型设计目标

考虑到 Wiki 托管环境无法方便直接管理数据库，本项目不建议前端直接读取 Excel 或数据库文件，而是先将数据整理为标准化 JSON。整理后的前端数据模型需要满足：

1. 可以直接通过 HTTP 拉取。
2. 数据结构清晰，前端无需做大量原始清洗。
3. 支持按敌人 ID 快速索引。
4. 支持按 buff ID 快速反查倍率与名称。
5. 支持前端生成 tip 链与显示规则。

### 5.5 Excel 到 JSON 的整理流程

推荐的数据整理流程如下：

#### 第一步：抽取敌人主数据

从 [`DB/整合表.xlsx`](DB/整合表.xlsx) 中提取每个敌人的：

- 名称
- 主索引 ID
- 基础 HP
- 基础躯干
- 基础躯干恢复
- 天色序列
- 阶段数
- 兵种信息
- 各类 buff 起始 ID
- 特殊模式相关字段

生成 `enemies.json`。

#### 第二步：抽取 buff 明细数据

整理所有会参与计算或 tip 展示的 buff，至少包含：

- buff ID
- buff 名称
- hp 倍率
- stamina 倍率
- stamina recover 倍率
- 所属类别
- 备注或显示名

生成 `buffs.json`。

#### 第三步：抽取界面元数据

整理前端固定选项与枚举，包括：

- 天色枚举
- 难度枚举
- 模式枚举
- 阶段标签生成规则

生成 `ui-meta.json`。

#### 第四步：预生成显示规则数据

根据敌人字段生成显示判断所需的结构，例如：

- 是否 boss
- 是否心中敌人
- 是否怨恨之鬼
- 是否允许显示再战
- 是否允许显示死斗
- 是否允许显示连战
- 是否应隐藏钟鬼和双难
- 是否可能发生普通/钟鬼合并

生成 `display-rules.json`，或并入 `enemies.json`。

### 5.6 字段清洗、映射与校验要求

在数据整理阶段应完成以下处理：

1. ID 字段统一为数值或统一字符串格式，不可混用。
2. 空值字段显式标注为 `null`，而不是空字符串。
3. 所有 buff 起始 ID 应验证其可否展开为对应序列。
4. 特殊敌人的模式字段必须单独校验。
5. 对于 [`special_buff`](Doc/敌人数据结构和计算逻辑.md:34) 中 HP 与躯干倍率不同的情况，必须在输出结构中分开保存。
6. 对于 tip 所需名称字段，整理阶段应补全，而不是运行时拼接猜测。

## 6. 前端数据模型设计

### 6.1 敌人主表结构

建议使用 `enemies` 作为核心数据对象，按主索引 ID 建立字典。每个敌人建议包含以下字段：

```json
{
  "id": 14004200,
  "name": "苇名流 水生氏成",
  "npcParamId": 14004200,
  "enemyType": "elite",
  "isBoss": true,
  "baseHp": 714,
  "baseStamina": 123,
  "baseStaminaRecover": 45,
  "phaseNum": 2,
  "timeSeq": [0, 1, 2, 3],
  "baseBonusBuffStart": 7140,
  "phaseChangeBuff": {
    "2": 291350
  },
  "specialBuff": 291400,
  "mindBattlefieldBuffStart": 7950,
  "reBattleBuffStart": 7954,
  "deathBattleBuffStart": 7958,
  "enemyTypeBuff": 7902,
  "ngBuffStart": 7800
}
```

实际字段名可在实现阶段调整，但项目书应先明确语义边界。

==标注==：这里有一个错误，只有enemytype为boss的才是boss，所以上面示例数据的isboss字段值应该是false

### 6.2 buff 索引表结构

建议将所有 buff 统一收敛为 `buffs` 字典，以 ID 作为 key，value 中包含：

- 名称
- 类别
- hp 倍率
- stamina 倍率
- stamina recover 倍率
- 显示备注

这样前端既可用于计算，也可用于 tip 展示。

### 6.3 场景与显示规则结构

为避免前端反复从原始字段做大量逻辑判断，建议加入 `displayRules` 概念，保存如下信息：

- `hideBellAndHard`
- `showRebattle`
- `showGauntlet`
- `showDeath`
- `innerOnlyModes`
- `mergeDifficultyWhenBaseBonusLength2`

### 6.4 tip 明细结构

tip 输出建议在前端计算完成时生成统一结构：

```json
{
  "metric": "hp",
  "finalValue": 17811,
  "steps": [
    {
      "type": "base",
      "label": "基础hp",
      "id": null,
      "value": 714
    },
    {
      "type": "buff",
      "label": "区域天色加成|苇名城 主城 (第二次入侵) _ 默认",
      "id": 7144,
      "value": 4.25
    }
  ]
}
```

这样既适合前端渲染，也适合后续导出为字符串。

### 6.5 建议的数据文件拆分方式

建议至少拆分为以下文件：

- `enemies.json`
- `buffs.json`
- `ui-meta.json`
- `version.json`

若显示规则相对复杂，可追加：

- `display-rules.json`

## 7. 计算引擎方案

### 7.1 输入参数模型

前端计算引擎每次计算时的输入参数建议包括：

- `enemyId`
- `metric`，取值为 `hp`、`stamina`、`staminaRecover`
- `mode`，取值为主线、再战、死斗、修罗连战
- `difficulty`，取值为普通、钟鬼、交符、双难
- `timeSeq`
- `ngNum`
- `phase`

### 7.2 计算流程总览

建议的计算流程如下：

1. 根据 `enemyId` 读取敌人主数据。
2. 根据 `metric` 选取对应基础值。
3. 解析当前模式下需要参与计算的 buff 集合。
4. 解析 [`base_bonus_buff`](Doc/敌人数据结构和计算逻辑.md:20) 及其派生序列。
5. 根据天色、难度、周目决定实际应调用的 buff ID。
6. 叠加阶段切换 buff 与特殊 buff。
7. 完成全部乘算后统一向下取整。
8. 同步生成 tip 链路。

### 7.3 buff 解析逻辑

计算引擎中的 buff 解析层应至少包含：

1. 起始 ID 转完整列表。
2. 根据难度与天色决定索引位。
3. 判断列表缺失时是否回退到普通逻辑。
4. 处理 [`base_bonus_buff`](Doc/敌人数据结构和计算逻辑.md:165) 长度为 2 的特殊合并情况。
5. 处理再战/死斗/修罗连战的顺序重映射。

### 7.4 数值乘算链设计

每次计算都建议先生成一个“乘算步骤数组”，再计算最终结果。原因有二：

1. 方便 tip 展示。
2. 方便排查计算错误。

乘算步骤应按业务语义顺序保留，而不是仅仅保存乘积结果。

### 7.5 特殊分支处理

计算引擎必须显式处理以下分支：

1. 一周目与多周目的周目加成差异。
2. 普通/钟鬼与交符/双难的分支差异。
3. [`special_buff`](Doc/敌人数据结构和计算逻辑.md:34) 的多倍率字段差异。
4. [`phase_change_buff`](Doc/敌人数据结构和计算逻辑.md:29) 仅在对应阶段生效。
5. 心中敌人、怨恨之鬼等特殊显示/模式规则。

### 7.6 tip 文案生成逻辑

tip 文案应由结构化步骤自动生成，格式目标可参考：

```text
【基础hp|714】*【区域天色加成|苇名城 主城 (第二次入侵) _ 默认|7144|4.25】*【多周目平衡|苇名城 主城 (第二次入侵) _ 默认|7644|1.55】*【周目加成|8及以上交符|1.9】*【兵种加成|兵种特效_精英|7902|1.125】*【特殊加成|水生氏成红眼|291400|2】
```

即：

- 基础项显示名称与数值
- buff 项显示类别、名称、ID、倍率
- 最终按乘法顺序拼接

## 8. 页面结构与交互方案

### 8.1 页面整体结构

根据 [`sumary/frontdemo_design_summary.md`](sumary/frontdemo_design_summary.md)，页面建议采用以下区块：

1. 顶部说明区
2. 控制区
3. 说明区
4. 主表格区
5. tip 详情区

### 8.2 控制区设计

控制区建议包含以下控件：

- 敌人下拉
- 周目下拉
- 天色下拉
- 阶段下拉
- 可选模式切换

其中下拉位置与整体排布应参考 [`DB/表格布局示例.xlsx`](DB/表格布局示例.xlsx) 的结构逻辑。

### 8.3 结果表格设计

主表格区为本页面核心，需满足：

1. 表头层级清晰。
2. 不同模式列可按规则动态显示或隐藏。
3. 难度行可按规则合并显示。
4. 数值单元格可点击。
5. 外层带横向滚动容器。

### 8.4 tip 栏设计

tip 栏应放在主表格下方或侧边，承担以下职责：

1. 展示当前点击单元格对应的完整计算链。
2. 展示当前敌人、周目、天色、难度、模式、阶段信息。
3. 清晰区分基础值、场景 buff、周目 buff、兵种 buff、特殊 buff。

### 8.5 交互反馈与降级策略

建议采用以下交互策略：

1. 默认进入页面时加载一个合理的初始敌人。
2. 条件变化后立即刷新结果表格。
3. 当某列不适用于当前敌人时直接隐藏，而非置灰占位。
4. 小屏设备允许横向滚动，不强行压缩表格结构。

## 10. 技术架构与部署方案

### 10.1 总体架构

推荐采用“外部静态前端 + 外部静态数据 + Wiki 页面嵌入”的总体架构。

架构流程如下：

1. 使用 [`DB/整合表.xlsx`](DB/整合表.xlsx) 作为源数据。
2. 通过整理流程导出标准化 JSON。
3. 将前端页面与 JSON 一并部署到外部静态托管平台。
4. 在 BWiki / [`MediaWiki`](Doc/项目需求.md:9) 页面中嵌入该前端应用。

### 10.2 数据托管方案

根据 [`Doc/项目需求.md`](Doc/项目需求.md:13)，Wiki 内部上传数据库不便，因此建议将整理后的 JSON 数据与静态页面托管在免费且稳定的平台，例如：

1. GitHub Pages
2. 配合 jsDelivr 作为静态资源加速源
3. 或其他可稳定提供静态文件访问的免费平台

项目书中的推荐主张是：

- 前端与数据文件统一使用静态托管
- 不依赖动态数据库服务
- 数据更新走版本化发布流程

### 10.3 前端嵌入方案

嵌入方式建议预留两种：

1. `iframe` 嵌入
2. 若宿主支持，则通过页面脚本加载外部前端模块

首选 `iframe` 的原因是：

- 宿主隔离性更强
- 样式冲突更少
- 对托管页面权限要求更低

### 10.4 缓存与更新策略

为避免数据更新后用户仍命中旧缓存，建议：

1. 数据文件带版本号或查询参数。
2. 保留 `version.json` 用于前端检测当前数据版本。
3. 每次正式数据更新后同步更新版本信息。
4. 重要修正发布后进行人工校验。

### 10.5 稳定性与维护方案

为保证长期可维护，建议：

1. 原始 Excel 作为数据编辑源。
2. JSON 作为发布产物，不直接手工维护。
3. 数据整理流程固定化。
4. 前端计算逻辑与数据结构保持解耦。
5. 特殊敌人逻辑集中管理，避免散落在多个组件中。

## 13. 风险与后续扩展

### 13.1 当前风险点

1. 原始 Excel 中个别字段可能存在空值、例外或人工录入不一致。
2. 特殊敌人规则较多，若前期未集中整理，后期容易出现隐性 bug。
3. 托管平台可能存在缓存或跨域限制。
4. 若未来规则继续扩展，前端逻辑复杂度会持续上升。

### 13.2 应对策略

1. 在数据整理阶段增加校验清单。
2. 将特殊敌人和特殊模式规则集中建模。
3. 采用版本化 JSON 与可追踪发布流程。
4. 保持计算逻辑与显示逻辑分层。

### 13.3 后续扩展方向

本项目完成后，可在相同架构上继续扩展：

1. 基础数据展板
2. 伤害计算器
3. 搜索与筛选增强
4. 结果分享链接
5. 多语言名称映射与资料辅助展示

## 15. 结论

本项目的本质不是单纯做一个网页表格，而是将 [`Doc/敌人数据结构和计算逻辑.md`](Doc/敌人数据结构和计算逻辑.md) 中复杂且零散的敌人数值规则，转化为一个适用于 BWiki / [`MediaWiki`](Doc/项目需求.md:9) 托管场景的、可维护、可解释、可持续更新的静态数据应用。

从实施角度看，本项目最关键的成功因素有三个：

1. 先完成数据整理与结构化建模。
2. 再实现可解释的前端计算引擎。
3. 最后以适配 Wiki 的轻量样式与静态托管方案完成部署。

按本项目书执行，可以将现有文档、Excel 数据和页面风格参考，系统化沉淀为后续开发的统一依据。
